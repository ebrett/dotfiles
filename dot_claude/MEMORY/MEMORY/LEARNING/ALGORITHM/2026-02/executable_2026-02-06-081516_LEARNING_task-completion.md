**What to improve next time:**

1. **Multiple version systems create confusion:** PAI has three distinct versioning concepts: PAI System version (2.5), Algorithm version (0.2.25), and SKILL.md generation timestamp. Users see all three and don't know which is which. The status line helps by showing "PAI: v2.5 ALG: v0.2.25", but initial confusion is understandable.

2. **Settings file has redundant version fields:** Both `paiVersion` at root level and `pai.version` nested field track the same thing. During upgrades, both must be updated or they get out of sync. This is a maintenance burden.

3. **Documentation comments can contradict actual values:** The `_docs._overview` field said v2.5 while version fields said 2.4. When documentation and data disagree, users don't know which to trust. Version should be single source of truth, not duplicated in comments.

4. **SKILL.md hardcodes algorithm version:** The SKILL.md header shows "v0.2.24" but the authoritative `LATEST` file is v0.2.25. SKILL.md is autogenerated, so this suggests the generation template needs updating to read from LATEST file instead of hardcoding.

5. **Upgrade process needs version field updates:** When upgrading PAI, the settings.json version fields must be updated as part of the upgrade flow. This wasn't done automatically, leaving the system in a partially-upgraded state.
